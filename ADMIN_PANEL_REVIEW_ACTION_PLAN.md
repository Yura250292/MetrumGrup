# Admin Panel Review Action Plan

Дата перевірки: 2026-04-22

## Контекст

Цей документ стосується лише внутрішньої адмінки.

Публічний сайт і клієнтський кабінет тут не оцінюються.
Ціль адмінки на поточному етапі: бути стабільним робочим інструментом для команди.

## Коротко

Основа адмінки вже сильна:

- є великий функціональний контур
- є окремий `admin-v2`
- є дашборди, проєкти, кошториси, ресурси, чат, фінансові модулі

Але для щоденної внутрішньої роботи є 5 ключових зон, які варто виправити першочергово:

1. не працює lint
2. адмін-навігація перевантажена декоративним стилем
3. архітектура змішана між `admin` і `admin-v2`
4. чат та окремі внутрішні екрани ще сирі по UX
5. візуальна система адмінки занадто “маркетингова” для бек-офісу

## Пріоритети

### P0 — критично

#### 1. Полагодити технічний контроль якості

Проблема:

- `npm run lint` зараз не працює
- це прибирає базовий запобіжник від регресій

Де:

- [package.json](/Users/admin/Igor-Shiba/metrum-group/package.json:10)

Чому це важливо:

- форми, таблиці, права доступу і server/client межі в адмінці легко ламаються непомітно
- без лінту росте технічний борг
- команді важче безпечно вносити дрібні зміни

Що зробити:

- перевести lint на робочу конфігурацію для Next.js 16
- додати явний ESLint config у репозиторій
- окремо перевірити правила для:
  - hooks
  - accessibility
  - `no-img-element`
  - unused imports
  - server/client boundaries
- додати команду, яка гарантовано проходить локально і в CI

Definition of done:

- `npm run lint` стабільно запускається
- команда бачить реальні помилки, а не збій конфігурації

### P1 — високий пріоритет

#### 2. Спрощення стилю sidebar і навігації

Проблема:

- sidebar в адмінці виглядає занадто декоративно
- кольори секцій не мають чіткої робочої семантики
- активні стани занадто “шумні”

Де:

- [AdminSidebar.tsx](/Users/admin/Igor-Shiba/metrum-group/src/components/layout/AdminSidebar.tsx:100)

Чому це важливо:

- адмінка має читатись швидко
- оператори проводять у ній багато часу
- інтерфейс повинен зменшувати втому, а не підсилювати її

Що зробити:

- прибрати pearlescent-градієнти з навігації
- залишити 1 спокійний активний стан для всіх груп
- звести кольори до функціональної ролі:
  - нейтральний інтерфейс
  - один primary accent
  - окремі кольори лише для статусів/ризиків
- посилити контраст тексту й зменшити декоративні тіні

Definition of done:

- sidebar виглядає стримано й читабельно
- активний пункт помітний без візуального перевантаження

#### 3. Звести адмінку до одного маршрутизованого шару

Проблема:

- зараз існує legacy `/admin` і новий `/admin-v2`
- логіка переходів частково тримається на redirect

Де:

- [next.config.ts](/Users/admin/Igor-Shiba/metrum-group/next.config.ts:12)

Чому це важливо:

- це ускладнює дебаг
- це ускладнює підтримку коду
- це створює зайвий ментальний шум для команди

Що зробити:

- визначити `admin-v2` як єдиний основний адмінський інтерфейс
- скласти список legacy сторінок, які ще реально потрібні
- або мігрувати їх у `admin-v2`, або прибрати з активного використання
- поступово скоротити кількість redirect-правил

Definition of done:

- адмінка має один канонічний шлях розвитку
- legacy-шар не є частиною щоденної навігації команди

#### 4. Довести чат до робочого інструмента, а не каркаса

Проблема:

- базова структура чату є
- але UX ще виглядає сирим для щоденного використання

Де:

- [page.tsx](/Users/admin/Igor-Shiba/metrum-group/src/app/admin/chat/page.tsx:5)
- [ChatLayout.tsx](/Users/admin/Igor-Shiba/metrum-group/src/components/chat/ChatLayout.tsx:14)

Чому це важливо:

- внутрішній чат має економити час команди
- зараз він більше схожий на базовий модуль, ніж на завершений робочий канал

Що зробити:

- покращити empty-state
- додати швидкі стартові дії:
  - нова розмова
  - швидкий перехід до останніх діалогів
  - пошук по розмовах
- перевірити мобільну і desktop логіку списку/треду
- додати більш виразні стани:
  - unread
  - loading
  - no results
  - failed to load

Definition of done:

- чат має очевидний стартовий сценарій
- користувач не “зависає” на пустому екрані

### P2 — важливо, але після стабілізації

#### 5. Заспокоїти всю візуальну мову адмінки

Проблема:

- адмінка успадковує стилістичні рішення з публічного продукту
- через це деякі екрани виглядають занадто “промо”, а не “операційно”

Де видно:

- sidebar
- KPI-блоки
- темізація
- деякі header/hero секції

Чому це важливо:

- бек-офісу потрібна чітка ієрархія
- візуальний акцент має бути на:
  - діях
  - статусах
  - попередженнях
  - ключових числах

Що зробити:

- визначити окрему дизайн-систему саме для адмінки
- обмежити glow, неонові тіні, сильні градієнти
- уніфікувати:
  - card surfaces
  - border system
  - spacing
  - heading scale
  - table styles
  - form controls
- переглянути темну тему так, щоб вона була комфортною для довгого використання

Definition of done:

- адмінка виглядає як робочий продукт
- інтерфейс тримається на системі, а не на окремих ефектах

## Рекомендований порядок робіт

### Етап 1

1. Полагодити lint
2. Підчистити sidebar
3. Прибрати з навігації сирі або другорядні розділи

### Етап 2

1. Зафіксувати `admin-v2` як головну адмінку
2. Скоротити залежність від legacy redirect-шару
3. Уніфікувати основні layout-патерни

### Етап 3

1. Допрацювати чат
2. Уніфікувати empty/loading/error states
3. Вирівняти таблиці, фільтри, дії, header-блоки

## Практичний висновок

Зараз головне для адмінки не “додати ще більше можливостей”, а зробити її стабільнішою, простішою і спокійнішою в щоденній роботі.

Найкращий найближчий ефект дадуть:

- робочий lint
- спрощення sidebar
- фіксація `admin-v2` як єдиної адмінки
- доведення чату і внутрішніх робочих екранів до зрілого UX
