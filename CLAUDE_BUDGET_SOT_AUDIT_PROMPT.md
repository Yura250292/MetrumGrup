# Prompt для Claude: перевірка реалізації `Single Source of Truth` для бюджету

Скопіюй увесь текст нижче і відправ Claude.

---

Тобі треба провести аудит реалізації в цьому репозиторії щодо плану:

- `BUDGET_SINGLE_SOURCE_OF_TRUTH_PLAN.md`
- за потреби як додатковий контекст: `FINANCING_PROJECTS_INTEGRATION_IMPROVEMENT_PLAN.md`

## Завдання

Перевірити, що з цього плану вже реалізовано в коді, що реалізовано частково, а що ще не реалізовано.

Не пиши абстрактно. Потрібен саме code audit по репозиторію.

## Головна тема перевірки

Потрібно перевірити, чи система вже перейшла до моделі:

- бюджет живе в одному головному місці
- інші цифри є похідними
- одна й та сама сума не рахується двічі

Особливо перевір:

1. Чи `ProjectStageRecord` фактично став canonical source для бюджету
2. Чи `Project.totalBudget` став лише derived/cache значенням
3. Чи `FinanceEntry(source=PROJECT_BUDGET)` не дублює stage-based бюджет
4. Чи `FinanceEntry(source=STAGE_AUTO)` використовується як derived projection, а не як окреме незалежне джерело правди
5. Чи summary / dashboard / budget-vs-actual / project finance рахують бюджет за одним правилом
6. Чи є захист від подвійного підрахунку `PROJECT_BUDGET + STAGE_AUTO`
7. Чи є захист від подвійного підрахунку `PROJECT_BUDGET + ESTIMATE_AUTO`
8. Чи є явне правило пріоритету між `STAGE_AUTO`, `ESTIMATE_AUTO`, `PROJECT_BUDGET`
9. Чи budget reset до `0` уже працює правильно
10. Чи сторінка проєкту показує бюджет як суму stage layer, а не як окрему ручну цифру

## Як саме працювати

1. Прочитай файл `BUDGET_SINGLE_SOURCE_OF_TRUTH_PLAN.md`
2. Знайди всі релевантні місця в коді:
   - schema / prisma models
   - finance summary
   - budget-vs-actual
   - project endpoints
   - stages endpoints
   - sync endpoints
   - mirror-service
   - financing queries
   - project UI, якщо там є окрема логіка сум
3. Для кожного правила з плану дай статус:
   - `Реалізовано`
   - `Частково реалізовано`
   - `Не реалізовано`
4. Для кожного статусу обов’язково наведи:
   - конкретні файли
   - конкретні рядки
   - коротке пояснення
5. Якщо логіка суперечлива, прямо це напиши:
   - де саме одна частина системи вже працює по-новому
   - а де інша ще живе по-старому

## Що не можна робити

- не відповідати загальними словами
- не переказувати план без перевірки коду
- не писати “ймовірно” без посилання на код
- не пропускати місця, де ще є legacy логіка

## Формат відповіді

Відповідь побудуй строго в такій структурі:

### 1. Executive Summary

Коротко:

- наскільки реалізація вже відповідає плану
- чи справді вже є одне джерело правди для бюджету
- чи ризик подвійного підрахунку ще лишився

### 2. Статус по кожному ключовому правилу

Для кожного пункту окремо:

#### Rule: `<назва правила>`

- Статус: `Реалізовано / Частково реалізовано / Не реалізовано`
- Докази:
  - `[path:line]` що саме знайдено
  - `[path:line]` що саме знайдено
- Висновок:
  - коротке пояснення

Перевір мінімум такі правила:

- `ProjectStageRecord` є canonical source
- `Project.totalBudget` є derived/cache
- `PROJECT_BUDGET` не дублює `STAGE_AUTO`
- існує єдина budget-layer policy
- summary не сумує кілька budget layers одночасно
- budget-vs-actual не конфліктує з project/stage logic
- budget reset до `0` працює коректно
- project page і financing page показують одну й ту ж логіку бюджету

### 3. Де ще залишився ризик подвійного підрахунку

Окремим списком:

- де саме ще можуть одночасно враховуватись кілька шарів
- в яких сценаріях це станеться
- які модулі найбільш ризикові

### 4. Що вже можна вважати завершеним

Короткий список того, що реально вже зроблено і працює по плану.

### 5. Що ще не дороблено

Короткий список того, що ще треба зробити, щоб модель “single source of truth” справді запрацювала до кінця.

### 6. Точкові наступні кроки

Дай 5-10 конкретних наступних технічних кроків у порядку пріоритету.

## Додаткова вказівка

Якщо знайдеш, що в коді вже є частина нової логіки, але поруч лишилась стара legacy-логіка, то це не вважай “повністю реалізовано”. У такому випадку статус має бути `Частково реалізовано`.

Якщо знайдеш, що один модуль працює по stage-driven budget, а інший ще по `PROJECT_BUDGET`, це треба явно винести як архітектурну неузгодженість.

Фінальна відповідь має бути суворою і технічною, не дипломатичною.
