# ADMIN V2 UX/UI Simplification Plan

## Мета

Зменшити складність `admin-v2` без втрати функціоналу.

Ціль не в тому, щоб прибрати можливості, а в тому, щоб:

1. показувати систему через сценарії користувача, а не через внутрішню структуру модулів
2. знизити cognitive load на щоденних екранах
3. зробити перші дії очевидними для нових користувачів
4. розділити:
   - огляд
   - операційну роботу
   - аналітику
   - довідники та рідкісні інструменти

---

## Ключові проблеми зараз

### 1. Перевантажена інформаційна архітектура

У sidebar багато великих груп і дублюючих за змістом входів:

- `Фінансування`
- `Фінансовий облік`
- `Звіти`
- `Звіти виконробів`
- `Постачальники`
- `Стратегічне планування`
- `AI Кошторис`
- `Усі кошториси`
- `Накладні (скан)`

Для системи це різні модулі.
Для користувача це одна предметна область, яка зараз розірвана на окремі точки входу.

Наслідок:

- нова людина не розуміє, куди йти
- досвідчена людина витрачає час на перемикання між екранами
- одна й та сама робота розбивається на кілька маршрутів

### 2. Головний dashboard перевантажений

Головна сторінка намагається бути одночасно:

- executive summary
- operational control center
- AI briefing
- KPI board
- project health screen
- team analytics

У результаті dashboard виглядає інформативно, але не веде до дії.

Наслідок:

- складно зрозуміти, що робити прямо зараз
- немає одного очевидного блоку пріоритетів дня
- операційна і аналітична функція змішані

### 3. Фінансовий модуль перевантажений вкладками

Усередині `Фінансування` зараз змішані:

- огляд
- операції
- бюджет/факт
- погодження
- скани
- календар
- архів
- pivot
- табелі

Це занадто багато для одного екрана верхнього рівня.

Наслідок:

- модуль виглядає “важким”
- складно зрозуміти основний порядок роботи
- вторинні інструменти конкурують із щоденними

### 4. Форми показують внутрішню модель системи

У фінансових формах і flow користувач бачить речі типу:

- `PLAN / FACT`
- `INCOME / EXPENSE`
- `entryIntent`
- `costType`
- `costCodeId`

Для внутрішньої моделі це нормально.
Для UX це неправильно.

Користувач мислить не моделлю даних, а наміром:

- додати витрату
- додати борг
- додати надходження
- погодити чек
- прив'язати постачальника

### 5. AI / OCR / імпорт входять занадто глибоко в фінальну бухгалтерську логіку

AI зараз місцями створює майже фінальний запис.
Правильніше, щоб AI створював структуровану чернетку, а не “готову істину”.

Наслідок:

- менеджер перевіряє вже фінальний об'єкт замість заявки
- вища ймовірність семантичних помилок
- важче пояснити людині, що відбулося

### 6. Візуально частина робочих екранів все ще виглядає як “сильний концепт”, а не як операційний інструмент

Проблема не в тому, що UI поганий.
Проблема в тому, що для data-dense роботи іноді забагато рівноцінних блоків, акцентів, станів і вкладок.

Наслідок:

- очі не ведуться до головного
- важливі дії не домінують над другорядними
- система виглядає “широкою”, а не “керованою”

---

## Принципи нового UX

### 1. Сценарій важливіший за модуль

Показуємо користувачу не “де лежить сутність у системі”, а:

- що він хоче зробити
- що потребує уваги
- який наступний крок

### 2. Один екран = одна головна роль

Кожен top-level screen повинен відповідати на одне питання:

- що сьогодні треба зробити
- що відбувається у проєктах
- які є фінансові дії
- що чекає погодження

Не можна змішувати на одному рівні:

- огляд
- повний реєстр
- довідники
- архів
- глибоку аналітику

### 3. Progressive disclosure

Спочатку просте.
Потім деталізація.

Порядок подачі:

1. summary
2. CTA / next action
3. короткий список критичних об'єктів
4. фільтри
5. детальна таблиця
6. архів / розширені режими

### 4. Намір дії замість технічної термінології

Не `PLAN / FACT`.
Не `entryIntent`.
Не “створити finance entry”.

А:

- `Додати витрату`
- `Додати борг`
- `Додати надходження`
- `Внести кошторис`
- `Погодити заявку`

### 5. AI = асистент структурування, не джерело істини

AI має:

- витягнути текст
- розкласти позиції
- припустити постачальника
- припустити статтю

Але фінальне рішення:

- борг чи оплачено
- який контрагент
- яка стаття бюджету

приймає людина.

### 6. Операційні екрани мають бути спокійнішими

Для щоденної роботи пріоритети:

- контраст
- щільність
- ієрархія
- читабельність

Не пріоритети:

- декоративність
- надлишок badge-ів
- зайві візуальні стани

## Нова структура dashboard

## Ціль

Зробити dashboard стартом робочого дня, а не універсальною аналітичною стіною.

### Нова послідовність блоків

#### Блок 1. `Потребує уваги сьогодні`

Показувати тільки те, що потребує реакції:

- прострочені задачі
- задачі на сьогодні
- непогоджені заявки виконробів
- прострочені платежі
- проєкти з ризиком

Це має бути головний блок.

#### Блок 2. `Мої дії`

Для конкретного користувача:

- мої задачі
- мої погодження
- нові згадки
- нові повідомлення

#### Блок 3. `Стан бізнесу`

Короткі KPI:

- активні проєкти
- очікувані надходження
- борги постачальникам
- cash position

#### Блок 4. `Що змінилось`

- нові платежі
- завершені задачі
- оновлення по проєктах
- нові заявки виконробів

#### Блок 5. `Аналітика`

Сховати нижче або винести в окремий режим:

- stage analytics
- team pulse
- AI summary
- breakdowns

### Нові режими dashboard

Не один універсальний режим, а 3:

1. `Операції`
2. `Керівник`
3. `Аналітика`

#### Операції

Для менеджера / адміністратора.
Фокус на задачах, погодженнях, блокерах і діях.

#### Керівник

Для owner / super admin.
Фокус на KPI, грошах, ризиках, динаміці.

#### Аналітика

Для глибших графіків, breakdowns, trends.

### Що прибрати з першого екрану

- другорядні декоративні hero-подачі
- великі summary-карти без next action
- блоки, які лише “гарно пояснюють”, але не ведуть до роботи

## Нова структура фінансового модуля

## Поточна проблема

`FinancingView` зараз намагається бути одночасно:

- home
- ledger
- approvals center
- scan center
- budget analytics
- payment calendar
- archive
- pivot workspace

### Нова модель верхнього рівня

У фінансах має бути лише 3 головні режими:

1. `Огляд`
2. `Операції`
3. `Погодження`

### Secondary actions

Інші режими не повинні бути основними вкладками першого рівня:

- `Скани`
- `Календар`
- `Архів`
- `Pivot`
- `Табелі`

Їх треба перенести:

- або у меню `Ще`
- або в окремі підмаршрути
- або відкривати як інструменти з `Операцій`

### Рекомендована структура фінансів

#### 1. Огляд

Показує:

- бюджет
- зобов'язання
- факт
- cash summary
- червоні зони

Не показує повний реєстр одразу.

#### 2. Операції

Головний робочий реєстр:

- витрати
- надходження
- борги
- оплати
- фільтри
- масові дії

#### 3. Погодження

Окремий операційний центр:

- заявки виконробів
- скани
- імпортовані чернетки
- записи, що вимагають уточнення

#### 4. Аналітика

Окремий screen або secondary route:

- budget vs actual
- pivot
- category breakdown
- period comparison

### Головна кнопка у фінансах

Одна основна кнопка:

- `Додати`

При кліку відкривається простий menu:

- `Витрата`
- `Надходження`
- `Борг`
- `Оплата постачальнику`
- `Кошторис`
- `Скан або імпорт`

Не треба показувати користувачу сирі технічні категорії.

## Новий UX для ручних форм

## Поточна проблема

Форма питає модель даних.
Має питати бізнес-подію.

### Нова схема першого кроку

Замість набору технічних полів перший крок форми:

`Що саме ви хочете внести?`

Варіанти:

1. `Витрата вже оплачена`
2. `Є борг постачальнику`
3. `Надійшли гроші від клієнта`
4. `Нам мають заплатити`
5. `Планова сума бюджету`

### Після вибору змінюється форма

#### Якщо `Витрата вже оплачена`

Поля:

- проєкт
- дата
- сума
- назва
- категорія
- постачальник або одержувач
- стаття витрат
- файл / чек

Система сама ставить:

- `ACTUAL_EXPENSE`

#### Якщо `Є борг постачальнику`

Поля:

- проєкт
- дата
- сума
- назва
- постачальник
- стаття витрат
- документ

Система сама ставить:

- `COMMITTED_EXPENSE`

#### Якщо `Надійшли гроші від клієнта`

Поля:

- проєкт
- дата
- сума
- клієнт
- опис
- файл

Система сама ставить:

- `ACTUAL_INCOME`

#### Якщо `Нам мають заплатити`

Поля:

- проєкт
- дата
- сума
- клієнт
- причина / документ

Система сама ставить:

- `COMMITTED_INCOME`

#### Якщо `Планова сума бюджету`

Поля:

- проєкт
- категорія
- стаття
- сума

Система сама ставить:

- `BUDGET_INCOME` або `BUDGET_EXPENSE`

### Що прибрати з ручної форми верхнього рівня

- `PLAN / FACT`
- `type` як технічне поле
- `entryIntent`
- необов'язкові поля, які не потрібні для вибраного сценарію

## Новий UX для AI / OCR / імпорту

## Правильна роль AI

AI не має створювати фінальний фінансовий запис одразу.
AI має створювати:

- чернетку
- заявку
- parsed candidate

### Новий flow

#### Крок 1. Завантаження

Користувач:

- додає фото
- PDF
- Excel
- текст

#### Крок 2. AI розбір

AI пробує визначити:

- дату
- суму
- контрагента
- позиції
- категорію
- можливу статтю витрат

#### Крок 3. Людське підтвердження

Менеджер або користувач бачить не фінальний ledger, а просту форму:

- що знайшов AI
- що не розпізнав
- що треба вибрати вручну

#### Крок 4. Бізнес-рішення

Людина відповідає:

- це оплачено чи борг
- хто контрагент
- до якої статті віднести

#### Крок 5. Лише після цього створюється фінансовий запис

### Переваги

- менше помилок
- простіше пояснювати логіку
- однакова модель для manual / import / AI

## Новий UX для виконроба

## Ціль

Виконроб має бачити максимально простий flow, без бухгалтерської мови.

### Простий сценарій

1. Вибрати об'єкт
2. Додати витрату
3. Написати текстом або прикріпити фото
4. Перевірити, що AI розклав позиції
5. Відправити менеджеру

### Що має бачити виконроб

Не повинен бачити:

- бюджет
- ledger
- складні довідники
- зобов'язання vs факт у технічних термінах

Повинен бачити:

- проєкт
- список своїх заявок
- статус:
  - `чернетка`
  - `на перевірці`
  - `потрібно уточнення`
  - `підтверджено`
  - `відхилено`

### Форма виконроба

Поля:

- об'єкт
- дата
- текст витрати
- фото / чек
- сума, якщо відома

Опційно:

- кількість
- одиниця
- коментар

### AI допомога

Після введення тексту або фото:

- AI пропонує назви позицій
- AI розбиває по item-ах
- AI пропонує тип витрати
- AI пробує знайти постачальника

Але виконроб не приймає фінальне бухгалтерське рішення.

---

## Новий UX для менеджера по заявках виконробів

## Ціль

Менеджер не повинен “розшифровувати систему”.
Він має просто приймати короткі рішення.

### На екрані review менеджер має бачити

Для кожної позиції:

- назва
- сума
- тип витрати
- AI впевненість
- запропонований постачальник
- фото / чек
- запропонована стаття

### Які рішення приймає менеджер

1. `Підтвердити / відхилити`
2. `Вибрати постачальника`
3. `Вибрати статтю витрат`
4. `Позначити як:`
   - `Борг`
   - `Оплачено`

### Перед approve має бути summary

Наприклад:

- `3 позиції підуть у борг постачальникам`
- `2 позиції підуть у фактичні витрати`
- `1 позиція без статті витрат`

Це важливо для довіри і контролю.

### Мова кнопок

Не:

- `Approve`
- `Set intent`

А:

- `У борг`
- `Оплачено`
- `Підтвердити`
- `Повернути на уточнення`

## Role-based UX

## Проблема

Система вже має role model, але UX ще не достатньо role-first.

### SUPER_ADMIN / OWNER

Потрібно:

- фінансовий огляд
- ризики
- KPI
- статус проєктів
- стратегічні блоки

Не потрібно на першому екрані:

- детальна операційна дрібнота

### MANAGER

Потрібно:

- задачі
- погодження
- проблемні проєкти
- заявки виконробів
- оперативні фінансові дії

### HR

Потрібно:

- люди
- ролі
- задачі
- активність

Не потрібні фінансові важкі блоки.

### FOREMAN

Потрібно:

- мої об'єкти
- додати витрату
- мої заявки
- статус рішень

### CLIENT

Потрібно:

- свій проєкт
- прогрес
- комунікація
- документи

### Висновок

Треба не просто ховати пункти по ролях.
Треба змінювати:

- головний сценарій
- пріоритети блоків
- CTA
- структуру стартового екрана

---

## Візуальні принципи оновлення

### 1. Менше рівноцінних акцентів

На екрані має бути:

- одна primary action
- 1-2 secondary actions
- решта тихі

### 2. Таблиці мають починатись із короткого operational summary

Перед великою таблицею:

- скільки записів
- скільки потребують дії
- скільки з помилками

### 3. Badge-и використовувати економно

Badge має означати важливий стан:

- `прострочено`
- `на погодженні`
- `борг`
- `оплачено`

Не варто badge-ами маркувати все.

### 4. Менше “важких” hero-блоків у робочих зонах

Hero добре працює:

- на старті модуля
- на onboarding
- на пояснювальних сторінках

Hero гірше працює:

- у щоденних операційних реєстрах

### 5. Вища щільність там, де це робочий екран

Для операційних сторінок допустимо:

- менші відступи
- компактніші таблиці
- спокійніший фон
- слабші тіні

## Конкретний roadmap впровадження

## Phase 1. IA cleanup

### Ціль

Спочатку спростити маршрути і назви, не переписуючи всю логіку.

### Що робимо

1. Перебудувати sidebar
2. Згрупувати фінансові entry points
3. Перенести рідкісні модулі в `Ще`
4. Вирівняти назви пунктів меню під людську мову

### Результат

- менше top-level шуму
- простіше орієнтуватись

## Phase 2. Dashboard refactor

### Що робимо

1. Переписати порядок блоків
2. Вивести `Потребує уваги сьогодні` у верхню частину
3. Додати role-based dashboard modes
4. Винести глибоку аналітику нижче або окремо

### Результат

- dashboard веде до дії
- менше перевантаження на вході

## Phase 3. Finance module simplification

### Що робимо

1. Скорочуємо top-level tabs до:
   - `Огляд`
   - `Операції`
   - `Погодження`
2. `Pivot`, `Календар`, `Архів`, `Табелі`, `Скани` переводимо в secondary routes
3. Робимо одну головну кнопку `Додати`

### Результат

- фінансовий модуль стає зрозумілішим
- менше “панелі керування космічним кораблем”

## Phase 4. Intent-based forms

### Що робимо

1. Прибираємо з першого кроку технічні поля
2. Вводимо сценарії:
   - оплачена витрата
   - борг
   - надходження
   - очікуваний прихід
   - бюджет
3. Автоматично мапимо це в `financeNature`

### Результат

- менше помилок
- простіше навчати людей

## Phase 5. AI/OCR/import unification

### Що робимо

1. Всі scan/import/AI flows переводимо в `чернетка -> перевірка -> підтвердження`
2. Забираємо пряме створення фінального запису там, де ще потрібна людська семантика
3. Уніфікуємо manager review

### Результат

- одна логіка для manual / import / AI
- менше семантичних збоїв

## Phase 6. Foreman UX polish

### Що робимо

1. Спрощуємо інтерфейс виконроба
2. Підсилюємо review flow для менеджера
3. Додаємо summary перед confirm
4. Покращуємо статуси і пояснення

### Результат

- швидший ввід
- менше двозначностей
- краща синхронізація з новою фінансовою моделлю

## Phase 7. Visual cleanup

### Що робимо

1. Зменшуємо кількість декоративних станів
2. Прибираємо зайві glow/shadow ефекти на операційних екранах
3. Підсилюємо контраст і ієрархію
4. Робимо компактніші data screens

### Результат

- система виглядає доросліше
- легше працювати довго

## Очікуваний результат

Після цих змін система має відчуватись так:

- новий користувач швидше розуміє, куди йти
- менеджер швидше робить щоденні дії
- фінансовий блок читається як процес, а не як складний реєстр
- виконробу простіше внести дані
- AI допомагає, а не ускладнює відповідальність
- інтерфейс стає більш операційним і менш когнітивно важким
