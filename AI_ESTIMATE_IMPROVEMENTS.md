## 1. Що було виправлено

### 1.1 Множники якості (КРИТИЧНА ПРОБЛЕМА)

**Було:**
| Tier | Material | Labor |
|------|----------|-------|
| economy | x1.0 | x1.0 |
| standard | x1.0 | x1.0 |
| premium | x1.05 | x1.0 |
| luxury | x1.10 | x1.0 |

**Різниця economy-luxury = 10%. Тому ціни завжди "приблизно однакові".**

**Стало:**
| Tier | Material qty | Material price | Labor price |
|------|-------------|---------------|-------------|
| economy | x0.95 | x0.75 | x0.85 |
| standard | x1.0 | x1.0 | x1.0 |
| premium | x1.08 | x1.40 | x1.25 |
| luxury | x1.15 | x2.20 | x1.60 |

**Різниця economy-luxury = ~3x для матеріалів, ~2x для робіт.**

**Файли:** `modifiers.ts`, `base-agent.ts`

### 1.2 Catalog Provider — вибір бренду по якості

**Було:** Завжди бере `brands[0]` або `averagePrice` — незалежно від якості.

**Стало:** `selectBrandByQuality()` вибирає бренд відповідно до `qualityTier`:
- economy → economy бренд (наприклад, Сталекс 340₴)
- premium → premium бренд (наприклад, ArcelorMittal 420₴)

**Файл:** `price-engine/providers/catalog.ts`

### 1.3 Прокинуто qualityTier через весь pipeline

- `PriceQuery` тепер має поле `qualityTier`
- Всі 10 агентів передають `context` в `enrichWithPriceEngine()`
- `ZeroPriceFixer` отримує `budgetRange` з wizard
- `MasterEstimateAgent` дає AI чіткі інструкції по бюджетному класу

## 2. Аналіз ефективності обробки даних

### 2.1 Обробка зображень

| Аспект | Поточний стан | Проблема |
|--------|--------------|----------|
| Розмір | Повна роздільність | Немає компресії/ресайзу |
| Кодування | Base64 (buffer.toString('base64')) | +33% розміру |
| Паралелізм | Послідовна обробка | Одне фото за раз |
| Промпт | Базовий: стан, тип, матеріали | Не витягує деталі конструкцій |
| Кешування | Є (vectorizer, file dedup) | Працює добре |

**Рекомендація:** Компресувати до 1024px, конвертувати в WebP, обробляти 3-5 фото паралельно.
**Економія:** ~1000-2000 токенів на фото.

### 2.2 Обробка PDF

| Аспект | Поточний стан | Проблема |
|--------|--------------|----------|
| Підхід | "Витягни ВЕСЬ текст" — brute-force | Немає фільтрації сторінок |
| Великі PDF | Обрізка до 50k символів | Але все одно відправляє все |
| Повторне використання | RAG кешує чанки | Працює добре |

**Рекомендація:** Спочатку визначити тип документа (5 перших сторінок), потім витягувати тільки релевантні розділи.
**Економія:** ~50,000 токенів на великий PDF.

### 2.3 Чанкінг (vectorizer)

| Аспект | Поточний стан | Проблема |
|--------|--------------|----------|
| Розмір чанку | 512 "слів" | Не токенів! 512 укр. слів ≈ 665 токенів |
| Overlap | 50 слів (9.7%) | Надлишковий |
| Метод | Split by whitespace | Наївний, не по реченнях |

**Рекомендація:** Перейти на токен-based чанкінг (256 токенів, 20 overlap). Фільтрувати порожні/шаблонні чанки.

### 2.4 Prozorro інтеграція

| Аспект | Поточний стан | Проблема |
|--------|--------------|----------|
| API виклики | 3-5 на кошторис | Висока латентність (15-50с) |
| Кешування | 24h для завершених тендерів | Немає крос-проектного кешу |
| Якість | Без фільтрації по якості | Змішує economy і premium тендери |

**Рекомендація:** Кешувати пошукові запити крос-проектно. Додати якісну фільтрацію.

## 3. Аналіз витрат токенів

### 3.1 Поточна структура витрат ($0.50-0.80 за кошторис)

```
Pre-Analysis:        ~$0.02  (1 виклик OpenAI)
Master Agent:        ~$0.45  (13 секцій × ~$0.035)
Zero-Price Fixer:    ~$0.05  (1-2 виклики fallback)
Prozorro enrichment: ~$0.03  
──────────────────────────────────────
РАЗОМ:               ~$0.55
```

### 3.2 Головне марнування токенів

| Проблема | Токени | $ | Пріоритет |
|----------|--------|---|-----------|
| **Master context в КОЖНУ секцію** | ~19,500 | $0.20 | КРИТИЧНО |
| **Wizard data дублюється** | ~6,500 | $0.07 | ВИСОКИЙ |
| **gpt-4o для ВСІХ секцій** | ~10,000 | $0.15 | ВИСОКИЙ |
| Gemini Google Search не трекається | ~5,000 | $0.03 | СЕРЕДНІЙ |
| PDF brute-force parsing | ~50,000 | $0.03 | СЕРЕДНІЙ |
| **РАЗОМ марнується:** | | **~$0.48** | |

### 3.3 Відсутній моніторинг

Зараз НЕМАЄ логування витрат токенів. Неможливо:
- Трекати реальну вартість кожного кошторису
- Виявляти аномальні витрати
- Оптимізувати на основі даних

## 4. Рекомендації з покращення

### Фаза 1: Швидкі виграші (1-2 тижні, -30-40% витрат)

#### 4.1 Кешувати master context
**Файл:** `master-estimate-agent.ts:736-739`

Зараз masterContext (1500-2000 токенів) копіюється у кожну з 13 секцій = ~20K зайвих токенів.

**Рішення:** Винести ключові факти проекту в коротке summary (200-300 токенів) для повторних секцій. Повний контекст давати лише першій секції.

**Економія:** ~$0.15-0.20 за кошторис.

#### 4.2 Диференціювати моделі по секціях
**Файл:** `master-estimate-agent.ts:540, 602`

Зараз gpt-4o для всіх секцій. Прості секції (Оздоблення, Сантехніка) можуть працювати на gpt-4o-mini.

| Секції для gpt-4o (складні) | Секції для gpt-4o-mini (прості) |
|-|-|
| Фундамент, Демонтаж, Електрика | Оздоблення, Вентиляція, Сантехніка |
| Стіни, Дах | Протипожежна безпека |

**Економія:** ~$0.10-0.15 за кошторис.

#### 4.3 Додати логування токенів
**Всі агенти:**
```typescript
const response = await openai.chat.completions.create({...});
console.log(`[tokens] ${agent}: in=${response.usage.prompt_tokens} out=${response.usage.completion_tokens}`);
```

### Фаза 2: Оптимізація обробки даних (2-3 тижні, -10-15% додатково)

#### 4.4 Компресія зображень перед Gemini Vision
**Файл:** `rag/vectorizer.ts:302`

Додати ресайз до 1024px і конвертацію в WebP перед відправкою.

#### 4.5 Розумне парсінг PDF
**Файл:** `pdf-helper.ts:40`

Замість "Витягни ВЕСЬ текст":
1. Перші 5 сторінок → визначити тип документа
2. Якщо технічна специфікація → сторінки 2-10
3. Якщо креслення → лише metadata

#### 4.6 Токен-based чанкінг
**Файл:** `rag/vectorizer.ts:224-236`

Замінити word-based (512 слів) на token-based (256 токенів) з js-tiktoken.

### Фаза 3: Продвинуті оптимізації (3-4 тижні)

#### 4.7 Стрімінг замість batch
Генерувати секції стрімінгом, щоб зменшити output token waste від max_tokens=8000 (середнє використання: 2000-4000).

#### 4.8 Semantic cache для Prozorro
Кешувати результати пошуку крос-проектно з семантичним ключем (а не точним текстом).

#### 4.9 Кешування файлів в пам'яті
**Файл:** `r2-client.ts:127-144`

Додати in-memory кеш (5-хв TTL) для файлів, які завантажуються з R2 під час генерації.
