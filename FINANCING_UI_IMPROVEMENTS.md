# FINANCING_UI_IMPROVEMENTS

## Мета

Зробити блок `Фінансування` простим, швидким і зрозумілим для щоденної роботи.

Користувач повинен за 10-20 секунд отримувати відповіді на 3 головні питання:

1. Що зараз відбувається з грошима?
2. Де є проблема або відхилення?
3. Яку дію треба зробити прямо зараз?

---

## Поточна проблема

Зараз блок `Фінансування` технічно сильний, але UX побудований навколо структури даних:

- `PLAN / FACT`
- `INCOME / EXPENSE`
- окремо `Платежі`

Для системи це логічно. Для людини в роботі це не найзручніший підхід.

Основні мінуси поточного UX:

- головний екран мислить квадрантами, а не задачами користувача
- слабкий набір фільтрів у UI порівняно з тим, що вже підтримує backend
- є плутанина між `Платежі` і `Фінансування`
- немає блоку `потребує уваги`
- немає швидкого сценарію для типових операцій
- немає saved views / quick presets
- немає наочного зв’язку `план -> факт`
- форма хороша як CRUD, але слабка для швидкого щоденного внесення даних

---

## Продуктова ціль

Потрібно перетворити `Фінансування` з реєстру операцій на робочий фінансовий центр.

Цей блок має вирішувати 4 сценарії:

1. Швидко внести нову операцію.
2. Швидко знайти потрібні записи.
3. Швидко побачити ризики і прострочення.
4. Швидко зрозуміти стан грошей по проєкту або по компанії.

---

## Рекомендована нова структура сторінки

### 1. Верхня панель керування

Повинна містити:

- `Project switcher`
- `Період`
- `Пошук`
- `Фільтри`
- `Збережені види`
- `Швидке додавання`
- `Експорт`

### 2. Основні вкладки сторінки

- `Огляд`
- `Операції`
- `Платіжний календар`
- `Архів`

### 3. Блок `Огляд`

Показувати не 4 квадранти як головний фокус, а коротку управлінську картину:

- факт баланс
- плановий баланс
- фактичні доходи
- фактичні витрати
- планові доходи
- планові витрати
- кількість відкритих планових записів без факту
- кількість операцій без вкладень

### 4. Блок `Потребує уваги`

Окремий список проблемних точок:

- планова дата минула, а факту немає
- перевитрата відносно плану
- витрата без чека / без рахунку
- великий платіж найближчими днями
- операція без проєкту
- підозріло велика сума
- неузгоджені або непідтверджені записи

### 5. Блок `Операції`

Один центральний список-реєстр з можливістю:

- швидко перемикатись між типами записів
- групувати
- сортувати
- відкривати деталі
- редагувати прямо з таблиці або drawer

### 6. Блок `Платіжний календар`

Окремий view для планових надходжень і оплат:

- по днях
- по тижнях
- по місяцях
- з індикаторами прострочення
- з очікуваним cashflow

## Як має виглядати головний UX

### Режим 1: Огляд

Користувач відкриває сторінку і одразу бачить:

- поточний баланс
- очікуваний баланс до кінця періоду
- топ проблем
- найближчі великі витрати
- найближчі надходження
- проєкти з найбільшим відхиленням від плану

### Режим 2: Операції

Користувач працює з реєстром і має tabs/chips:

- `Усі`
- `План`
- `Факт`
- `Доходи`
- `Витрати`
- `Без проєкту`
- `З файлами`
- `Без файлів`
- `Прострочені`

### Режим 3: Платіжний календар

Користувач дивиться на рух грошей у часі:

- що має прийти
- що треба оплатити
- де касовий розрив

## Що треба змінити в UI

### 1. Прибрати квадранти як головний сценарій

Поточний `PLAN/FACT x INCOME/EXPENSE` grid можна залишити, але не як головну структуру сторінки.

Краще:

- винести його в окремий secondary view
- або лишити як компактний summary block

### 2. Зробити один головний реєстр операцій

Замість чотирьох окремих списків користувачу потрібен один список з фільтрами і швидкими перемикачами.

У рядку операції варто показувати:

- дата
- тип
- план / факт
- сума
- проєкт
- категорія
- контрагент
- автор
- вкладення
- статус / прапорець уваги

### 3. Зробити зрозуміле розділення `Платежі` і `Фінансування`

Потрібно прямо в UI пояснити різницю:

- `Платежі` = графік оплат по проєкту
- `Фінансування` = журнал планових і фактичних операцій

Добре рішення:

- підзаголовки
- короткі helper texts
- різні іконки
- різні візуальні акценти

### 4. Додати sticky summary

При скролі верхній summary має лишатися видимим:

- баланс
- факт
- план
- прострочено
- без підтвердження

### 5. Додати швидкі дії у верхню панель

- `+ Факт витрата`
- `+ Факт дохід`
- `+ План витрата`
- `+ План дохід`

Це має бути швидше, ніж відкривати потрібний квадрант і шукати кнопку всередині.

---

## Які фільтри потрібно додати

Backend уже підтримує більше, ніж показує UI. Це треба відкрити у фронті.

Обов’язкові фільтри:

- проєкт
- тип: `дохід / витрата`
- вид: `план / факт`
- категорія
- підкатегорія
- автор / відповідальний
- період
- є вкладення / немає вкладень
- архів / активні

Дуже бажані фільтри:

- тільки без проєкту
- тільки прострочені
- тільки великі суми
- тільки без контрагента
- тільки без опису
- тільки невідповідність плану і факту

### Швидкі пресети по даті

Замість постійного ручного вибору `from/to`:

- `Сьогодні`
- `7 днів`
- `30 днів`
- `Цей місяць`
- `Наступний місяць`
- `Квартал`
- `Рік`

---

## Saved Views

Потрібно додати збережені сценарії перегляду.

Мінімальний набір:

- `Мої витрати`
- `Без проєкту`
- `План без факту`
- `Без вкладень`
- `Останні 30 днів`
- `Прострочені`
- `Великі суми`

Це сильно скоротить кількість повторних кліків.

## Найкорисніші нові фічі

### 1. Plan vs Fact matching

Найважливіша функція.

Система має вміти показувати:

- який план уже закритий фактом
- який план ще не закритий
- де факт перевищив план
- де факт менший за план

Це можна зробити через:

- link між записами
- автоматичне match by project/category/date/amount
- ручне підтвердження користувачем

### 2. Прапорці якості запису

Для кожної операції варто показувати статус якості:

- є вкладення
- є контрагент
- є опис
- прив’язано до проєкту
- співставлено з планом

Це допоможе фінансовій дисципліні без складного контролю.

### 3. Recurring operations

Для повторюваних платежів:

- оренда
- зарплати
- сервісні витрати
- регулярні надходження

Потрібно:

- шаблон
- частота повторення
- автостворення наступного запису

### 4. Counterparty memory

При виборі контрагента система має запам’ятовувати:

- останні контрагенти
- типові категорії
- типові суми
- типові описи

Це пришвидшить введення операцій у рази.

### 5. Approval / verification state

Для записів варто ввести життєвий цикл:

- `Чернетка`
- `На погодженні`
- `Підтверджено`
- `Проведено`
- `Архів`

Це особливо корисно для великих витрат.

### 6. Attachment compliance

Треба додати окрему видимість:

- є чек / немає чека
- є рахунок / немає рахунку
- є договір / немає договору

Мінімум:

- іконка
- фільтр
- alert block

### 7. Cashflow timeline

Графік надходжень і витрат по часу:

- день
- тиждень
- місяць

Потрібен дуже простий варіант без складної аналітики:

- очікувані гроші
- заплановані списання
- чистий потік
- потенційний касовий розрив

### 8. Прив’язка до задачі або estimate

Це дуже корисне покращення саме для цього проєкту.

Фінансовий запис має мати можливість бути прив’язаним до:

- task
- estimate
- project milestone

Це дасть контекст:

- чому виникла витрата
- хто її створив
- до якої роботи вона відноситься

## Як покращити форму створення операції

Поточна форма нормальна, але її треба зробити швидшою для реального користування.

### Обов’язково покращити

- quick presets зверху
- автопідстановку останніх значень
- шаблони для типових операцій
- inline hints для категорій
- preview вкладених файлів
- drag and drop для файлів

### Додати у форму

- статус запису
- ознака `потрібен документ`
- зв’язок з task / estimate
- прапорець `повторювана операція`
- прапорець `важлива`

### Що треба спростити

- не змушувати щоразу вручну обирати одні й ті самі поля
- дати сценарій `внести за 15 секунд`

---

## Як покращити проектний блок `Платежі`

Проєктний блок `Платежі` зараз простий і зрозумілий, але занадто пасивний.

Що варто додати:

- найближчий платіж
- прострочені платежі
- очікуваний платіж цього тижня
- статусний фільтр
- швидке позначення `оплачено / частково оплачено / прострочено`
- timeline замість тільки таблиці
- індикатор ризику по оплаті

Було б корисно додати:

- `payment progress by stages`
- `next expected payment`
- `collection risk`

## Простий цільовий сценарій для користувача

### Менеджер

Хоче за хвилину зрозуміти:

- скільки вже витрачено
- скільки ще заплановано
- які проєкти проблемні
- де очікуються великі виплати

### Фінансист

Хоче:

- швидко вносити записи
- працювати з документами
- бачити непідтверджені витрати
- контролювати повноту даних

### Керівник

Хоче:

- короткий зрозумілий summary
- ризики
- відхилення від плану
- cashflow на короткий період

## Definition of Done

Блок `Фінансування` можна вважати успішно покращеним, якщо:

- користувач за 1 екран бачить стан грошей без детального ручного аналізу
- новий запис можна внести за 10-15 секунд
- потрібні записи шукаються через фільтри за 2-3 кліки
- прострочення і проблемні місця видно без ручного перегляду всіх списків
- різниця між `Платежі` і `Фінансування` очевидна
- по проєкту і по компанії легко перемикатися без втрати контексту
