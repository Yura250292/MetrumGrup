# План спрощення AI-кошторису

## Ціль

Зробити AI-кошторис:

- простим для менеджера;
- передбачуваним у роботі;
- прозорим у логіці;
- без зайвих режимів і псевдо-налаштувань;
- без інтеграції з Prozorro у цьому флоу.

---

## Головне рішення

### 1. Повністю прибрати Prozorro з AI-кошторису

Що це означає:

- забрати `checkProzorro` з UI;
- забрати `prozorroSearchQuery` з payload;
- забрати відображення Prozorro як джерела даних;
- прибрати Prozorro enrichment з generation flow;
- прибрати Prozorro-аналітику з результату AI-кошторису;
- не показувати користувачу жодних згадок про Prozorro у V2.

Причина:

- це ускладнює продукт;
- це не є критичним для базового сценарію “швидко отримати зручний кошторис”;
- для внутрішніх і fit-out кошторисів користь обмежена;
- це розмиває головний продукт і перевантажує інтерфейс.

---

## Продуктове бачення після спрощення

AI-кошторис має працювати так:

1. Користувач додає файли.
2. Користувач вибирає тип кошторису:
   - `Внутрішні роботи`
   - `Будівництво з нуля`
3. Користувач вказує площу або система визначає її з креслень.
4. За бажанням користувач проходить майстер уточнення.
5. Система робить преаналіз.
6. Система генерує кошторис.
7. Система показує результат, проблемні місця і дозволяє відредагувати / доповнити / зберегти.

Ключовий принцип:

> Менше вибору на вході, більше якості й контролю на виході.

---

## Основні проблеми поточної реалізації

### 1. UI обіцяє більше, ніж реально працює

- користувач бачить вибір моделі;
- бачить RAG / Prozorro / джерела;
- бачить “опціональні” параметри;
- але фактичний backend flow значно жорсткіший і менш варіативний.

### 2. Є зайві рішення для користувача

На старті користувач змушений думати про:

- модель;
- тип кошторису;
- майстер;
- площу;
- джерела;
- Prozorro.

Для простого сценарію це забагато.

### 3. Є розрив між UX і реальною логікою

- площа в UI виглядає необов’язковою;
- wizard продається як дуже короткий;
- деякі контролли існують, але не впливають на генерацію так, як очікує користувач.

### 4. Архітектура перевантажена

- є legacy flow;
- є V2 flow;
- є кілька шарів валідації;
- є післягенераційні латки;
- є зайві залежності від додаткових джерел.

---

## Цільовий UX

## Екран 1. Старт

Залишити тільки:

- завантаження файлів;
- поле площі;
- перемикач:
  - `Внутрішні роботи`
  - `Будівництво з нуля`
- кнопку `Уточнити проєкт`;
- кнопку `Згенерувати кошторис`.

Прибрати:

- вибір моделі;
- блок Prozorro;
- блок “джерела даних” у поточному вигляді;
- технічні маркетингові деталі про RAG на першому екрані;
- фальшивий або неактивний preview.

### Логіка

- якщо площа не введена і її не можна надійно дістати з креслень, кнопка генерації має вести до чіткої помилки;
- якщо wizard не проходили, це допустимо;
- wizard не має блокувати швидкий сценарій.

---

## Екран 2. Майстер

### Принцип

Майстер має бути не “довгою анкетою”, а коротким уточненням тільки того, що реально впливає на кошторис.

### Новий формат

Побудувати майстер навколо 3-5 блоків:

1. Тип об’єкта і стадія.
2. Геометрія та площа.
3. Інженерія.
4. Оздоблення.
5. Особливі умови.

### Що важливо

- перестати комунікувати “5 кроків”, якщо їх фактично більше;
- зробити кроки динамічними, але показувати користувачу реальну кількість;
- прибрати слабокорисні поля, які не дають помітного ефекту на якість.

---

## Екран 3. Преаналіз

### Ціль

Преаналіз має відповідати лише на 3 питання:

1. Чи достатньо даних для генерації?
2. Чи є суперечності?
3. Що варто уточнити до запуску?

### Що прибрати

- зайву “аналітичну красу” без дії;
- поля, які не допомагають прийняти рішення `генерувати / не генерувати`.

### Що залишити

- коротке резюме;
- 3-7 ключових зауважень;
- чіткий CTA:
  - `Виправити`
  - `Все одно згенерувати`

---

## Екран 4. Результат

### Показувати в першу чергу

- загальну суму;
- кількість секцій;
- кількість позицій;
- ризики / попередження;
- що було автоскориговано;
- де відсутня впевненість.

### Не перевантажувати

- не виносити технічні внутрішні сутності в перший план;
- не змішувати технічний engine report з основним кошторисом;
- окремо показувати “виправлено автоматично”, “потребує перевірки”, “без ціни”.

---

## План змін

## Етап 1. Прибрати зайве з UI

### Зробити

- прибрати вибір моделі генерації;
- прибрати блок Prozorro;
- прибрати `checkProzorro`;
- прибрати `prozorroSearchQuery`;
- прибрати показ “RAG / Prozorro / джерела” як окремого продуктового вибору;
- прибрати неактивний preview;
- привести тексти wizard до реального сценарію.

### Результат

Стартовий екран стане коротшим, зрозумілішим і чеснішим.

---

## Етап 2. Спрощення backend flow

### Зробити

- прибрати передачу `checkProzorro` у V2 flow;
- прибрати Prozorro enrichment із master/orchestrator flow;
- прибрати повернення `prozorroAnalysis` у відповіді V2;
- прибрати залежність V2 від Prozorro-related report data;
- залишити тільки:
  - документи;
  - wizard;
  - візуальний обмір;
  - внутрішню базу матеріалів;
  - rule-based validators;
  - zero-price fixer.

### Результат

Generation pipeline стане простішим, дешевшим і стабільнішим.

---

## Етап 3. Уніфікація правил

### Зробити

- визначити один канонічний validation pipeline для V2;
- не тримати паралельно кілька різних правил для схожих сценаріїв;
- чітко розвести:
  - `hard errors`
  - `warnings`
  - `auto-fixes`

### Результат

Команда буде краще розуміти, чому система відхилила або скоригувала кошторис.

---

## Етап 4. Скорочення wizard

### Зробити

- переглянути всі поля wizard;
- залишити лише ті, що реально змінюють:
  - склад секцій;
  - обсяги;
  - тип матеріалів;
  - інженерію;
  - клас оздоблення.

### Прибрати або сховати

- рідко корисні технічні деталі;
- поля, які майже не впливають на генерацію;
- надто “інженерні” розгалуження для менеджера.

### Результат

Менше тертя на вході, вищий completion rate, кращий UX.

---

## Етап 5. Чесна комунікація системи

### Зробити

- якщо площа потрібна — так і написати;
- якщо майстер опціональний — не занижувати його реальну довжину;
- якщо автокорекція змінила суми — показати це явно;
- якщо деякі ціни знайдено приблизно — маркувати це.

### Результат

Вища довіра до AI-кошторису.

---

## Що залишаємо як сильні сторони

- візуальний обмір креслень;
- master flow генерації;
- нормалізацію AI-чисел і AI-item parsing;
- rule-based validation;
- zero-price fixer;
- внутрішню базу матеріалів;
- можливість refine / supplement / manual edit після генерації.

---

## Що прибираємо повністю з цього продуктового сценарію

- Prozorro як джерело для AI-кошторису;
- Prozorro як елемент стартового UX;
- Prozorro як частину V2 explainability;
- будь-які проміжні UI-рішення, які лише демонструють “складність системи”, але не спрощують життя користувачу.

---

## Пріоритет виконання

### P1

- прибрати Prozorro з UI;
- прибрати вибір моделі;
- прибрати неактивний preview;
- привести тексти і readiness до реальної логіки.

### P2

- відрізати Prozorro з backend V2 flow;
- спростити response payload;
- прибрати зайві поля з result screen.

### P3

- скоротити wizard;
- уніфікувати validation rules;
- відшліфувати explainability.

---

## Очікуваний результат

Після спрощення AI-кошторис має сприйматися так:

- завантажив файли;
- вказав базові параметри;
- отримав нормальний чернетковий кошторис;
- швидко побачив слабкі місця;
- поправив і зберіг.

Не так:

- обирав між моделями;
- думав про Prozorro, RAG і внутрішню архітектуру;
- намагався вгадати, які налаштування реально працюють;
- губився між технічними станами системи.

---

## Наступний крок

Після погодження цього плану:

1. почати з UI-спрощення V2;
2. паралельно відрізати Prozorro з V2 backend pipeline;
3. окремо пройтись по wizard і скоротити його до мінімально корисного набору полів.
