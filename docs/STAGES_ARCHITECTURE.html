Етапи виконання — як побудовано План ↔ Факт

Це візуальний розбір: де лежать етапи в БД, як одна модель тримає плани і факти одночасно, як на них вішаються деталізовані фінансові проводки, і як UI рендерить чотири режими: Усі Тільки План Тільки Факт Порівняти.

★
TL;DR. В БД це одна таблиця — ProjectStageRecord — з двома наборами колонок (plan* і fact*), плюс копії published* для механіки draft → publish. Деталізовані суми лежать окремо у FinanceEntry, де тип записа визначає enum kind: PLAN | FACT. Дельта рахується на льоту як fact − plan.

2 Модель ProjectStageRecord — серце системи

Це і є той рядок у таблиці «Етапи виконання», який ви бачите в UI. Спільні поля (Назва, Відповідальний, Дати) лежать поряд; далі — симетричні колонки plan* і fact*, плюс заморожені published* для публікації клієнту.

Чому одна модель, а не дві? Бо «План» і «Факт» — це різні значення одного й того самого етапу. Якби розділити на дві таблиці, треба було б тримати JOIN на кожен запит, синхронізувати ключі й боротися з сиротами. Один рядок з парами колонок — простіше й швидше.

3 Модель FinanceEntry — деталізовані проводки

Поки ProjectStageRecord тримає зведення по етапу, FinanceEntry містить конкретні транзакції (накладні, табелі, акти). Кожна проводка явно позначена: це план чи факт, витрата чи дохід.

🟦 Запис типу PLAN

«Запланована витрата 50 000 ₴ на бетон для етапу Фундамент.» Створюється під час планування — ще до того, як гроші реально пішли.

🟩 Запис типу FACT

«Виплачено постачальнику 52 300 ₴ за бетон, накладна №…» Створюється коли подія сталася — наприклад, через звіт виконроба (source = FOREMAN_REPORT).

6 Як обчислюється агрегація

Зведення (скільки всього план/факт по етапу) рахує функція computeStageFinanceAggregates. Вона робить один groupBy по трьох колонках:

// Псевдокод
prisma.financeEntry.groupBy({
  by: ["stageRecordId", "kind", "type"],
  _sum: { amount: true },
  where: { projectId }
});

// На виході — 4 «корзини» на кожен етап:
selfPlanExpense  // kind=PLAN, type=EXPENSE
selfPlanIncome   // kind=PLAN, type=INCOME
selfFactExpense  // kind=FACT, type=EXPENSE
selfFactIncome   // kind=FACT, type=INCOME

// Похідні значення:
planResult = planIncome − planExpense
factResult = factIncome − factExpense
deltaExpense = factExpense − planExpense
deltaResult  = factResult  − planResult

8 Чому саме така архітектура — головні рішення

✅ Один рядок етапу, дві групи колонок

Плюс: один UPDATE замість двох; немає сиріт «план без факту»; легко малювати «Порівняти». Мінус: ширина таблиці, але це звичайна BI-практика.

✅ FinanceEntry зі спільним enum kind

Плюс: одна таблиця проводок — спільні індекси, фільтри, права доступу. Мінус: завжди фільтрувати по kind; зате SQL прозорий.

🟣 Окремі published* поля

Дозволяють редагувати draft не зачіпаючи опублікованої клієнтові версії. dirty-stages підсвічує що змінилося з моменту останньої публікації.

🟡 Дельта — НЕ зберігається

Δ обчислюється на льоту на сервері (агрегація) або в UI (для конкретних колонок). Нема ризику розсинхрону «джерело vs кеш».

9 Ментальна модель — у одному реченні

Етап — це один рядок з парами колонок plan* / fact* (+ published* копії).
Гроші по етапу — це багато рядків у FinanceEntry, розділених kind = PLAN | FACT, привʼязаних до етапу через stageRecordId.
Дельта — рахується. Її ніхто не зберігає.
