# AI_ASSISTANT_IMPROVEMENT_PLAN

## Мета

Зробити AI помічника не просто чат-вікном з відповідями, а надійним робочим помічником для щоденних задач у Metrum Group.

AI має:

1. Давати точні відповіді на основі реальних даних.
2. Безпечно виконувати дії в системі.
3. Пояснювати, що саме він зробив і чому.
4. Бути зручним у повсякденній роботі, а не лише в demo-сценаріях.

---

## Поточний стан

AI помічник уже має сильну базу:

- persisted conversations
- role-aware system prompt
- tool calling
- SSE streaming
- quick actions
- insights
- project context з URL
- write actions

Це вже хороший фундамент.

Але зараз AI сильніший як:

- інтерфейс до даних
- smart search по системі
- contextual assistant

ніж як:

- надійний операційний агент
- безпечний виконавець дій
- прозорий і передбачуваний інструмент для бізнес-користувача

---

## Основні проблеми

### 1. Write actions занадто прямі

AI уже вміє виконувати дії:

- create task
- update task
- assign task
- create project
- schedule payment
- record expense
- send notification

Але зараз для таких дій бракує окремого safe-flow.

Ризики:

- випадкове створення неправильних записів
- неправильні поля через неточний prompt
- небажані зміни без підтвердження користувача
- складність аудиту “що саме зробив AI”

### 2. Tool layer уже занадто великий

`tool-executors.ts` перетворюється на великий змішаний файл, де зібрані:

- проєкти
- задачі
- фінанси
- кошториси
- коментарі
- матеріали
- веб-пошук
- write actions

Це збільшує:

- складність підтримки
- ризик помилок
- складність permission control
- складність подальшого тестування

### 3. Контекст ще слабкий

Зараз project context передається через просту текстову вставку з URL.

Це корисно, але недостатньо.

AI повинен знати не лише `projectId`, а й короткий робочий контекст:

- назва проєкту
- статус
- поточний етап
- бюджет / сплачено
- кількість відкритих задач
- прострочення
- ризики

### 4. Немає прозорості по tools

Користувач зараз майже не бачить:

- які tools були викликані
- що повернув tool
- на основі яких даних зроблено висновок
- які дії реально були виконані

Для бізнес-інструменту це критично.

### 5. Немає quality loop

Система не збирає достатньо даних, щоб зрозуміти:

- де AI помиляється
- які questions часто не закриваються
- які tools дають помилки
- які prompts працюють гірше
- де AI робить зайві виклики

### 6. UX ще більше “чатовий”, ніж “робочий”

Помічник уже красивий і функціональний, але він ще не став частиною основного workflow.

Зараз це:

- sidebar assistant
- quick prompts
- streaming answer

Має стати:

- інструмент аналізу
- інструмент дії
- інструмент навігації
- інструмент прийняття рішень

---

## Продуктова ціль

AI помічник повинен працювати у двох режимах:

### 1. Режим аналізу

AI:

- відповідає
- шукає
- порівнює
- аналізує
- узагальнює
- пропонує наступні кроки

але нічого не змінює у системі.

### 2. Режим дії

AI:

- готує зміну
- показує preview
- чекає підтвердження
- виконує дію
- повертає результат
- пише audit trail

## Що потрібно покращити

## 1. Safe write-actions flow

### Проблема

AI не повинен одразу створювати проєкти, витрати чи платежі лише на основі одного текстового запиту.

### Потрібно зробити

Кожна write-action має проходити через 2 стадії:

1. `Draft action`
2. `Confirm action`

### Як це має працювати

AI отримує запит:

`Створи задачу по проєкту АТБ на демонтаж з дедлайном на п’ятницю`

Замість прямого виконання AI має показати:

- проєкт
- назву задачі
- дедлайн
- пріоритет
- кого буде призначено
- що буде записано в систему

І тільки після кнопки `Підтвердити` або явного підтвердження виконувати дію.

### Дії, які обов’язково мають бути confirm-only

- create project
- update project stage
- create task
- update task
- assign task
- schedule payment
- mark payment paid
- record expense
- add team member
- send notification

---

## 2. Розділення tools по доменах

### Проблема

Один великий `tool-executors.ts` буде швидко деградувати.

### Потрібно зробити

Розбити на модулі:

- `projects-tools.ts`
- `tasks-tools.ts`
- `finance-tools.ts`
- `estimates-tools.ts`
- `comments-tools.ts`
- `team-tools.ts`
- `external-search-tools.ts`
- `write-actions-tools.ts`

### Додатково

Потрібен окремий `tool-registry.ts`, де:

- описується tool metadata
- вказується required role
- вказується read/write type
- вказується requiresConfirmation

---

## 3. Tool schema validation

### Проблема

Зараз tool input занадто гнучкий.

Для AI це ризик:

- неправильний тип поля
- зайві поля
- відсутні required поля
- криві дати
- неочікувані статуси

### Потрібно зробити

Для кожного tool додати runtime validation, бажано через `zod`.

Переваги:

- безпечніше виконання
- чистіші помилки
- кращий DX
- легше тестувати

### Помилки мають бути user-friendly

Не просто:

- `invalid input`

А:

- `Для створення платежу потрібні сума і дата`
- `Статус задачі невалідний`
- `Проєкт не знайдено або недоступний`

---

## 4. Покращений контекст

### Поточна проблема

AI знає лише базовий project context з URL.

### Потрібно зробити

Перед кожним chat turn додавати компактний контекст пакетом:

- current page type
- current project
- current task
- current estimate
- user role
- швидкий operational summary

### Приклад context packet

- page: `project`
- projectId
- projectTitle
- status
- currentStage
- stageProgress
- tasksOpen
- overdueTasks
- unpaidPayments
- financeBalance

### Додатково

Для сторінок:

- tasks
- financing
- estimates
- reports

потрібно робити page-specific context.

## 5. Conversation memory і summary

### Проблема

Зараз історія просто накопичується.

Це погано для:

- токенів
- довгих розмов
- повторних звернень
- контекстної узгодженості

### Потрібно зробити

- summary після N повідомлень
- title regeneration
- pinned conversation
- rename conversation
- project-scoped memory
- last active project/topic memory

### Дуже корисно

Додати окремий тип пам’яті:

- `user preferences`
- `favorite project names`
- `common workflows`
- `preferred response style`

---

## 6. Tool transparency UI

### Проблема

Користувач повинен довіряти AI.

Без прозорості він не буде сприймати його як надійний інструмент.

### Потрібно зробити

У кожній assistant-відповіді мати expandable block:

- які tools були використані
- короткий результат tools
- які дані були знайдені
- чи була виконана write action

### Мінімальний варіант

- `Використано 2 джерела даних`
- `get_project_summary`
- `get_payment_status`

### Кращий варіант

Показувати timeline:

1. Знайдено проєкт
2. Отримано фінансові дані
3. Отримано статус платежів
4. Побудовано висновок

---

## 7. Page-aware quick actions

### Проблема

Quick actions зараз занадто загальні.

### Потрібно зробити

Quick actions повинні змінюватись залежно від сторінки.

### На project page

- `Покажи ризики цього проєкту`
- `Що тут прострочено`
- `Порівняй план і факт`
- `Хто зараз перевантажений`

### На tasks page

- `Що критичне сьогодні`
- `Кого треба перепризначити`
- `Які задачі заблоковані`

### На finance page

- `Покажи перевитрати`
- `Де касовий розрив`
- `Які платежі прострочені`

### На estimates page

- `Поясни структуру кошторису`
- `Що найдорожче`
- `Де основна маржа`

## 8. Action cards замість тільки тексту

### Проблема

AI зараз головним чином відповідає текстом.

### Потрібно зробити

Для типових ситуацій AI повинен повертати action cards:

- `Створити задачу`
- `Призначити виконавця`
- `Запланувати платіж`
- `Відкрити проєкт`
- `Перейти до фінансування`

### Перевага

Це переводить AI з режиму “розумна відповідь” у режим “робочий помічник”.

## 10. AI quality loop

Потрібен окремий процес покращення якості.

### Мінімум

- log conversation outcomes
- позначка `корисно / не корисно`
- тип помилки:
  - неправильні дані
  - слабкий аналіз
  - неправильна дія
  - не зрозумів намір
  - занадто довго

### Далі

- eval dataset з реальними питаннями
- regression checks на ключові сценарії
- окремі golden prompts для:
  - projects
  - tasks
  - finance
  - reports
  - onboarding/help mode

## 12. Режим рекомендацій

AI повинен не лише відповідати, а й підсвічувати корисні речі.

### Потрібно

- проактивні рекомендації
- короткі summary cards
- щоденні або контекстні suggestions

### Приклади

- `У вас 3 прострочені задачі в активних проєктах`
- `Є 2 проєкти з ризиком касового розриву`
- `У кошторисі цього проєкту найбільша стаття — бетон`

## 13. Role safety

### Проблема

AI має бути не просто role-aware у prompt, а role-safe на рівні execution.

### Потрібно

- чітко ділити tools по ролях
- окремо маркувати `read/write`
- окремо маркувати `internal-only`
- дублювати перевірку прав у кожному tool executor

### Особливо важливо

- client не повинен бачити внутрішні задачі
- engineer не повинен діставати зайві фінанси
- financier не повинен виконувати дії поза своєю зоною без перевірки

---

## 14. Search quality

## Definition of Done

AI помічник можна вважати якісно покращеним, якщо:

- він безпечно виконує дії лише після підтвердження
- користувач бачить, які дані були використані
- відповіді стабільно прив’язані до реального контексту сторінки
- assistant реально допомагає в роботі, а не лише “гарно відповідає”
- є вимірюваність якості, помилок і корисності
- AI інтегрований у ключові workflows: проєкти, задачі, фінанси, кошториси
