# Admin UI Improvement Plan

Дата: 2026-04-22

## Мета

Зробити адмінку:

- зручною для щоденної роботи
- візуально спокійною і професійною
- інформативною з першого погляду
- швидкою для сканування, а не лише “красивою”

## Базовий принцип

Цільовий стиль: `calm enterprise UI`

Це означає:

- спокійний фон
- нейтральні поверхні
- один основний accent color
- кольори лише для статусів, ризиків і дій
- сильна ієрархія контенту
- мінімум декоративних ефектів

## Ключова візуальна модель

### Що залишити

- сучасну чисту типографіку
- картки як будівельний блок системи
- чіткі status chips
- mobile shell і responsive layout
- токенізовану тему

### Що зменшити

- градієнти
- glow
- зайві кольорові плями
- надлишкові hero-блоки
- декоративні акценти в робочих зонах

### Що посилити

- читабельність
- щільність корисних даних
- швидкі сценарії дій
- однакові UI-патерни між екранами

## Пріоритети

### P1 — найближчі покращення

1. Sidebar
2. Header
3. Dashboard top area
4. Projects list views
5. Toolbar/filter pattern

### P2 — наступний етап

1. Finance screen simplification
2. Feed redesign
3. Unified table system
4. Empty/loading/error states

### P3 — системний polish

1. Design density modes
2. Saved views everywhere
3. Better compact mode patterns
4. Final visual cleanup

## План по зонах

## 1. Sidebar

### Проблема

Sidebar повинен бути інструментом навігації, а не найяскравішою частиною екрана.

### Що зробити

- зробити один універсальний active state
- прибрати декоративні тіні і надмірну кольоровість
- зменшити контраст background у навігаційних групах
- залишити кольорові бейджі тільки для дійсно важливих сигналів
- вирівняти вертикальний rhythm між пунктами

### Бажаний результат

- sidebar читається за 1-2 секунди
- активний розділ видно одразу
- навігація не відволікає від вмісту сторінки

### Рекомендована структура

- Дашборд
- Проєкти
- Кошториси
- Фінанси
- Ресурси
- Комунікація
- Налаштування

## 2. Header

### Проблема

Top bar зараз містить забагато елементів одного візуального пріоритету.

### Що зробити

- залишити в header тільки ключові глобальні елементи
- breadcrumbs зробити легшими і менш шумними
- page title / current context зробити сильнішим
- team avatars і частину utility actions винести в overflow або профільне меню
- theme toggle прибрати з центральної уваги

### Бажаний результат

- header виглядає як спокійна системна панель
- головний фокус переходить на вміст сторінки

## 3. Dashboard

### Ціль

Dashboard має бути стартовою робочою консоллю, а не просто набором красивих віджетів.

### Нова рекомендована структура

#### Блок 1. Потрібно сьогодні

- прострочені задачі
- прострочені платежі
- погодження, що чекають рішення
- дедлайни на сьогодні / найближчі 3 дні

#### Блок 2. Ключові показники

- активні проєкти
- cashflow / доходи / витрати
- нові кошториси
- завантаження команди

#### Блок 3. Наступні дії

- створити проєкт
- створити кошторис
- додати фінансову операцію
- відкрити критичний список

#### Блок 4. Операційний моніторинг

- projects at risk
- needs attention
- recent activity
- team load

### Що змінити в UI

- скоротити кількість KPI одного рівня
- менше декоративних chip-композицій
- сильніше виділити risk blocks
- зробити один явний блок “що робити зараз”

### Бажаний результат

- людина відкриває dashboard і одразу розуміє:
  - де є проблема
  - що треба зробити
  - куди йти далі

## 4. Projects

### Ціль

Сторінка проєктів повинна бути зручною і для огляду, і для керування.

### Що зробити

- додати режими:
  - Cards
  - Table
  - Compact
- зробити table view основним для desktop-користувачів
- залишити card view для огляду і візуального browsing
- compact mode зробити для великих списків

### У table view показувати

- назву
- клієнта
- менеджера
- статус
- етап
- бюджет
- оплачено
- залишок
- дедлайн
- ризик
- швидкі дії

### У card view залишити тільки потрібне

- назва
- статус
- етап
- 1-2 фінансові числа
- дедлайн
- ризик

### Бажаний результат

- великі списки проєктів легко скануються
- менеджер швидко порівнює записи між собою

## 5. Toolbar і Filters

### Проблема

На різних сторінках toolbar-и і filter blocks повинні виглядати як одна система.

### Єдиний патерн

Кожна велика сторінка має мати:

- title
- subtitle
- primary action
- search
- filters
- saved views
- export

### Що зробити

- стандартизувати порядок елементів
- винести secondary actions в overflow
- зробити filter chips над таблицею/контентом
- додати clear filters
- додати saved views для ключових screens

### Бажаний результат

- всі великі екрани мають одну логіку користування
- новому користувачу легше освоїти систему

## 6. Finance

### Проблема

Фінансовий екран насичений функціями і може перевантажувати.

### Що зробити

- спростити верхню зону екрана
- залишити одну головну primary action
- розділити “огляд” і “операції”
- зменшити кількість одночасно видимих control blocks
- згрупувати другорядні дії
- зробити більш явний compact режим для списків і таблиць

### Бажаний результат

- фінансовий модуль стає менш важким для сприйняття
- основні сценарії дій зрозумілі без навчання

## 7. Feed

### Ціль

Feed має бути не просто стрічкою, а інструментом швидкого реагування.

### Що зробити

- додати compact mode
- додати read/unread logic
- додати priority markers
- додати severity coloring для важливих подій
- додати quick actions з рядка
- додати sticky filters

### Бажаний результат

- feed допомагає приймати рішення
- важливі події не губляться серед другорядних

## 8. Tables

### Ціль

Створити один canonical table pattern для всієї адмінки.

### Що зробити

- sticky header
- row hover state
- row actions
- better sort model
- consistent filters
- bulk actions
- compact density mode
- mobile card fallback з однаковою структурою

### Бажаний результат

- усі таблиці поводяться однаково
- дані читаються швидше
- команда не вчиться щоразу новому патерну

## 9. Empty / Loading / Error States

### Що зробити

- empty state має пояснювати, що це за розділ
- empty state має вести до 1 головної дії
- loading має бути skeleton-based
- error має давати retry
- стани мають бути візуально однакові по всій системі

### Бажаний результат

- навіть “порожні” або “проблемні” стани виглядають зріло і допомагають користувачу

## 10. Типографіка і щільність

### Що зробити

- вирівняти scale заголовків
- зменшити кількість uppercase labels
- зробити консистентні стилі для:
  - page title
  - section title
  - helper text
  - labels
  - meta text
- додати два режими щільності:
  - comfortable
  - compact

### Бажаний результат

- UI виглядає професійніше
- на великих робочих екранах поміщається більше корисної інформації

## Швидкі wins

### За 1 спринт

1. Заспокоїти sidebar
2. Спрощення header
3. Переробити dashboard top section
4. Уніфікувати toolbar pattern
5. Додати table/compact view для проєктів

### За 2 спринт

1. Почистити finance top area
2. Додати canonical table behaviors
3. Переробити feed у більш операційний формат
4. Уніфікувати empty/loading/error states

## Definition of success

План можна вважати успішним, якщо після змін:

- адмінка виглядає спокійніше і професійніше
- користувачі швидше знаходять потрібні дії
- dashboard краще підказує пріоритети дня
- великі списки і таблиці легше скануються
- навігація й основні екрани мають єдину логіку

## Підсумок

Найкраще покращення для цієї адмінки дасть не додавання ще більшої кількості декоративного UI, а перехід до більш зрілого робочого інтерфейсу:

- менше шуму
- більше змісту
- сильніша ієрархія
- стабільні патерни
- чіткі дії

Головна ціль: щоб адмінкою було приємно користуватись щодня по багато годин, а не лише щоб вона добре виглядала на першому враженні.
