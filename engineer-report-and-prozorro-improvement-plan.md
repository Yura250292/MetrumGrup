# План покращення функції "Звіт інженера" та Prozorro-аналізу

## Мета

Перетворити поточний `Звіт інженера + Prozorro` з довідкового інструменту в робочий модуль для двох практичних задач:

1. **Оцінка ціни входу в торги**
   - бачити тендери, максимально схожі за бюджетом;
   - окремо бачити діапазон `±10%` від цільової суми;
   - розуміти, по якій ціні реально заходити в тендер;
   - оцінювати ризик демпінгу або завищення.

2. **Інженерний execution report**
   - не просто текстовий summary;
   - а зрозумілий, послідовний, практичний план:
     - що робити;
     - у якій послідовності;
     - на що звернути увагу;
     - де ризики;
     - що потрібно уточнити до старту;
     - які підготовчі дії обов’язкові.

- коротке пояснення, що було проаналізовано;
- опис, як формувався кошторис;

### Чого не вистачає

- немає реальної технологічної послідовності робіт;
- немає поділу на етапи;
- немає prep/start/critical dependencies;
- немає карти ризиків;
- немає списку необхідних уточнень перед стартом;
- немає окремого "плану дій для інженера".

## 1.3. Як зараз формується Prozorro-аналіз у pre-analysis

Поточний Prozorro flow у `PreAnalysisAgent`:

1. будується `searchQuery`;
2. генеруються multi-queries;
3. виконується повнотекстовий пошук через `searchTendersByText`;
4. результати дедупляться;
5. з БД підтягуються розпарсені кошториси `prozorroEstimateData`;
6. тендери збагачуються бюджетом, локацією і rough similarity;
7. рахується:
   - `topSimilarProjects`
   - `aggregatedLocations`
   - `priceDatabase`

## 2. Що саме потрібно для вашої задачі

Ти сформулював правильну прикладну задачу:

> потрібні дуже схожі тендери приблизно на такі суми `±10%`, щоб аналізувати приблизну ціну входу під час торгів.

Це означає, що система має давати не просто "схожі тендери", а окрему бізнес-відповідь:

- яка **цільова сума входу**;
- який **ринковий коридор**;
- які тендери **найближчі по бюджету і типу робіт**;
- по якій сумі **реально виграти**;
- де межа, нижче якої починається небезпечний демпінг.

Інженерний звіт теж має змінитися:

- із "описового summary"
- у "технічний робочий plan".

## 3. Основні проблеми поточної реалізації

## 3.1. Немає окремого режиму "ціна входу"

Зараз система:

- шукає релевантні тендери;
- показує topSimilarProjects;
- показує average price level.

Але вона не відповідає напряму на запит:

- `В який бюджетний діапазон нам заходити?`
- `Яка ставка виглядає реалістичною?`
- `Де безпечний мінімум?`

## 3.2. Бюджетний фільтр зараз занадто широкий

У manual search route використовується `±30%`.

Це добре для broad search, але погано для задачі bidding.

Для аналізу входу в торги потрібні щонайменше 3 кільця:

- `±10%` — ядро;
- `±20%` — розширене релевантне поле;
- `±30%` — контекстний резерв.

## 3.3. Similarity зараз не достатньо "тендерна"

Поточний matcher оцінює:

- бюджет;
- площу;
- CPV;
- keywords.

Цього недостатньо для bid decision.

Потрібно ще враховувати:

- завершені тендери vs активні;
- ціну переможця, а не лише очікувану вартість;
- тип замовника;
- регіон;
- дату;
- технічну складність;
- пакетність робіт;
- чи це повний обсяг, чи окремий підряд.

## 3.4. Engineer report занадто загальний

Поточний `analysisSummary` не дає:

- чіткої послідовності робіт;
- етапності;
- залежностей між роботами;
- checklists before start;
- resource/logistics notes;
- критичних точок приймання рішень.

## 3.5. Два Prozorro-пайплайни дублюють логіку

Зараз pre-analysis і manual Prozorro search працюють по-різному.

Це означає:

- різні результати для однієї задачі;
- складніше підтримувати;
- складніше пояснити користувачу, чому один модуль знайшов одне, а інший інше.

## 4. Якою має бути цільова модель

## 4.1. Новий формат інженерного модуля

Потрібно розділити модуль на 3 окремі блоки:

1. `Project Analysis`
2. `Tender Entry Analysis`
3. `Execution Plan`

### 1. Project Analysis

Що зрозуміли з проекту:

- тип об’єкта;
- площа;
- обсяг;
- тип робіт;
- критичні системи;
- ключові невизначеності;
- яких даних бракує.

### 2. Tender Entry Analysis

Що показує Prozorro:

- найближчі тендери по сумі;
- очікувана вартість;
- ціна переможця;
- знижка до очікуваної;
- регіон;
- тип замовника;
- дата;
- діапазон реального входу.

### 3. Execution Plan

Практична послідовність для інженера:

- підготовчі дії;
- що має бути погоджено до старту;
- рекомендований порядок виконання;
- критичні залежності;
- ризики;
- контрольні точки;
- що не можна запускати паралельно;
- що краще виносити в окремі пакети.

## 4.2. Prozorro повинен працювати як bid intelligence

Не просто "схожі тендери", а:

- `bid intelligence layer`

Тобто система має відповідати:

- який реалістичний ринок;
- де середина;
- де агресивний вхід;
- де небезпечний демпінг;
- де ми майже гарантовано будемо дорогими.

## 5. Що потрібно змінити

## 5.1. Уніфікувати Prozorro pipeline

Потрібно об’єднати поточні 2 режими в один сервіс:

- `searchTendersForEstimate()`

Він має:

1. отримувати estimate/wizard/project context;
2. будувати базовий broad search;
3. окремо формувати `tight budget matches`;
4. окремо формувати `market context matches`;
5. окремо формувати `winner price analysis`;
6. повертати normalized report object.

### Вихідні дані нового сервісу

```ts
type TenderEntryReport = {
  targetBudget: number;
  budgetBands: {
    tight10: TenderMatch[];
    near20: TenderMatch[];
    context30: TenderMatch[];
  };
  entryPrice: {
    recommended: number;
    aggressive: number;
    conservative: number;
    basedOnWinnersMedian: number;
    basedOnExpectedMedian: number;
  };
  winnerStats: {
    medianWinnerDiscountPercent: number;
    averageWinnerDiscountPercent: number;
    minWinnerDiscountPercent: number;
    maxWinnerDiscountPercent: number;
  };
  marketSignals: {
    regionFactor: string;
    dateFactor: string;
    competitionLevel: "low" | "medium" | "high";
  };
};
```

## 5.2. Додати окремий фільтр `±10%`

Це ключове покращення під твою задачу.

Потрібно завести 3 шари:

### `Core matches`

- тільки тендери в межах `±10%` від target budget;
- це основний блок для аналізу ціни входу.

### `Near matches`

- `±20%`;
- для розуміння ширшого поля.

### `Context matches`

- `±30%`;
- для ринку в цілому.

У UI це треба показувати окремо, а не змішувати.

## 5.3. Додати аналіз ціни переможця

Для торгів головне не просто очікувана вартість, а:

- `awardedAmount`
- відхилення `awardedAmount` від `expectedAmount`

Потрібно рахувати:

- median winner price;
- середню знижку від очікуваної ціни;
- типову глибину торгу;
- win corridor.

### Приклад корисного output

- Очікувана вартість по ядру тендерів: `31.2 млн`
- Медіана ціни переможця: `28.6 млн`
- Типова знижка переможця: `-8.3%`
- Рекомендована ціна входу: `28.4–29.1 млн`
- Агресивний демпінг нижче: `27.6 млн`

Саме це і є бізнес-корисний Prozorro insight.

## 5.4. Посилити scoring

Окрім поточних критеріїв, треба додати ваги:

- бюджетна близькість;
- winner price relevance;
- date freshness;
- region relevance;
- overlap по scope;
- type of buyer;
- complete vs active tender;
- parsed estimate availability.

### Рекомендований новий скоринг

- `budget proximity`: 35%
- `scope similarity`: 25%
- `winner price availability`: 10%
- `region similarity`: 10%
- `recency`: 10%
- `CPV relevance`: 10%

## 5.5. Переробити Engineer Report у structured output

Замість одного довгого тексту потрібно генерувати structured object:

```ts
type EngineerExecutionReport = {
  projectUnderstanding: string[];
  assumptions: string[];
  missingInputs: string[];
  executionSequence: Array<{
    stage: string;
    goals: string[];
    prerequisites: string[];
    risks: string[];
    controlPoints: string[];
  }>;
  tenderStrategy: {
    recommendedEntryRange: [number, number];
    comments: string[];
  };
};
```

Потім із цього можна:

- красиво рендерити UI;
- генерувати readable narrative;
- експортувати в PDF;
- використовувати для review.

## 5.6. Додати "Послідовність виконання"

Для інженера треба окремий блок:

- `Рекомендована послідовність робіт`

Наприклад:

1. Підтвердження вихідних даних та ревізія документації
2. Геодезія / підготовка майданчика
3. Підключення тимчасових мереж
4. Земляні роботи
5. Фундаменти
6. Коробка
7. Інженерні мережі
8. Оздоблення / благоустрій
9. Пусконалагодження
10. Фінальна здача

Але порядок має формуватись динамічно залежно від:

- нового будівництва;
- реконструкції;
- ремонту;
- оздоблення;
- комерційного чи житлового типу.

## 5.7. Додати "Що треба уточнити до старту"

Окремий блок:

- відсутні розміри;
- неясні вузли;
- немає геології;
- не підтверджено тип фундаменту;
- невідомий склад оздоблення;
- відсутня схема мереж;
- не визначені межі відповідальності суміжних підрядників.

Це дуже цінно для інженера і менеджера.

## 5.8. Додати "Тендерні ризики"

Окремий блок:

- ризик недооцінки логістики;
- ризик прихованих робіт;
- ризик неповної документації;
- ризик сезонності;
- ризик завищеного очікуваного бюджету;
- ризик агресивного ринку;
- ризик того, що в Prozorro референси з іншої категорії об’єктів.

## 8. Корисні додаткові ідеї

Нижче кілька сильних ідей, які можуть дуже підсилити цей напрямок.

## 8.1. Bid Recommendation Score

Додати фінальний сигнал:

- `Входити / Входити обережно / Не входити`

На основі:

- повноти документації;
- кількості ризиків;
- ринкового коридору;
- winner discount;
- валідності estimate.

## 8.2. Tender Risk Heatmap

Показувати ризики кольором:

- документація;
- геологія;
- логістика;
- сезонність;
- scope ambiguity;
- цінова невизначеність.

## 8.3. What-if pricing

Дати менеджеру 3 сценарії:

- базовий;
- агресивний;
- безпечний.

Наприклад:

- `Base entry`
- `Win-focused`
- `Margin-protected`

## 8.4. Regional tender intelligence

Аналіз по місту/області:

- де ціни нижчі;
- де агресивніший ринок;
- де замовники торгуються сильніше;
- де краща маржинальність.

## 8.5. Similarity by work package

Замість загальної схожості дати ще:

- схожість по фундаменту;
- по інженерії;
- по оздобленню;
- по благоустрою;
- по зовнішніх мережах.

Це дуже корисно, якщо проект не повний, а пакетний.

## 8.6. Competitor pattern intelligence

Якщо вдасться накопичувати історію:

- типові переможці;
- типові розміри знижки;
- типові сегменти, де виграють;
- категорії, де ринок перегрітий.

Це вже рівень реального тендерного advantage.

## 9. Практичний висновок

Поточна функція `Звіт інженера + Prozorro` уже має сильний фундамент:

- є pre-analysis;
- є пошук по тендерах;
- є top similar tenders;
- є агреговані локації;
- є price references;
- є окремий UI модуль.

Але зараз це ще:

- `reference + summary`

а потрібно зробити:

- `bid intelligence + engineering execution system`

Найбільший ефект дадуть 3 зміни:

1. окремий аналіз `±10%` для ціни входу;
2. аналіз `awardedAmount` та winner discount;
3. structured execution report із послідовністю робіт, ризиками і checklist.

Саме це перетворить модуль з "цікавого додатку" на робочий інструмент для прийняття рішень у торгах і запуску проекту.
