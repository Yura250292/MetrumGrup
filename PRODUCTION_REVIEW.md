# Production Review

Дата перевірки: 2026-03-31

## Коротко

Проєкт є Next.js 16 застосунком для Metrum Group, який у продакшні працює як:

- публічний маркетинговий сайт на `/`
- закритий адмін-кабінет на `/admin`
- закритий клієнтський кабінет на `/dashboard`
- набір API routes на Next.js App Router
- PostgreSQL + Prisma як основна база даних
- NextAuth credentials login з JWT-сесіями
- AI-маршрути для генерації, редагування й експорту кошторисів
- Vercel як цільовий runtime

Збірка `npm run build` проходить успішно.

Водночас у проєкті є кілька критичних проблем доступу до даних, кілька функціональних дефектів в AI-маршрутах та зламані перевірки якості перед деплоєм.

## Як система працює в продакшні

### Інфраструктура

- Деплой орієнтований на Vercel.
- У `vercel.json` вказано:
  - `framework: nextjs`
  - `buildCommand: prisma generate && next build`
  - регіон `fra1`
  - `maxDuration: 30` для API functions
- База даних: PostgreSQL через Prisma.
- Runtime:
  - публічні сторінки частково статичні
  - адмінка і кабінет клієнта здебільшого динамічні server-rendered сторінки
  - API routes працюють як serverless functions

### Авторизація і ролі

- Логін побудований на NextAuth credentials provider.
- Перевірка користувача:
  - пошук по email
  - перевірка `isActive`
  - звірка bcrypt password hash
- Сесія JWT, термін життя 30 днів.
- Основні ролі:
  - `SUPER_ADMIN`
  - `MANAGER`
  - `ENGINEER`
  - `FINANCIER`
  - `CLIENT`
  - `USER`

### Поведінка маршрутів

- `/`:
  - великий маркетинговий landing page
  - контент переважно хардкоджений у `src/app/page.tsx`
  - не виглядає як сторінка, що реально читає CMS-дані з БД
- `/login`, `/register`:
  - форми входу/реєстрації
- `/admin/*`:
  - внутрішній кабінет компанії
  - дашборд, проєкти, кошториси, матеріали, фінанси, користувачі, налаштування
- `/dashboard/*`:
  - клієнтський кабінет
  - проєкти клієнта, фото, платежі, документи

### Дані та бізнес-логіка

- Основні сутності:
  - users
  - projects
  - estimates
  - estimate sections/items
  - payments
  - photo reports
  - materials
  - workers/equipment/warehouse
  - CMS entities
  - notifications
  - audit logs
- Кошториси:
  - створюються вручну або через AI generation
  - проходять workflow review
  - можуть оброблятися фінансистом для рентабельності, логістики і податків
- AI-функції:
  - `generate`
  - `refine`
  - `export`
  - виконуються в межах HTTP request, без background queue

## Критичні проблеми

### 1. Витік даних через `/api/admin/projects`

Файл: `src/app/api/admin/projects/route.ts`

Проблема:

- `GET /api/admin/projects` перевіряє лише факт авторизації
- немає перевірки ролі
- будь-який авторизований користувач може отримати список усіх проєктів

Наслідки:

- клієнт або будь-який інший логінований користувач може отримати внутрішні проєкти
- витікають назви проєктів, статуси та дані клієнтів

Що виправити:

- додати обов'язкову role-check перевірку
- дозволити доступ лише:
  - `SUPER_ADMIN`
  - `MANAGER`
  - за потреби окремо `ENGINEER`/`FINANCIER`, якщо це справді потрібно бізнесом
- винести перевірку ролей у спільний helper

Пріоритет: Critical

### 2. Неправильна модель доступу в клієнтських API

Файли:

- `src/lib/auth-utils.ts`
- `src/app/api/projects/route.ts`
- `src/app/api/projects/[id]/payments/route.ts`
- `src/app/api/projects/[id]/photos/route.ts`

Проблема:

- `scopeByClient()` обмежує тільки роль `CLIENT`
- для будь-якої іншої ролі повертає порожній `where`
- роль `USER` фактично може бачити всі проєкти
- у деяких API перевірка ownership також виконується тільки для `CLIENT`

Наслідки:

- недостатньо контрольований доступ до фінансових та проєктних даних
- ролі, які не повинні читати всі дані, фактично це можуть

Що виправити:

- перейти від моделі "обмежуємо лише CLIENT" до моделі "явно дозволяємо конкретні ролі"
- визначити ACL для кожного API
- заборонити role `USER` доступ до внутрішніх проєктних endpoint'ів, якщо це не бізнес-вимога
- створити helper типу:
  - `requireRole([...])`
  - `requireProjectAccess(projectId, session)`

Пріоритет: Critical

### 3. AI generation route жорстко вимагає Gemini ключ навіть для OpenAI/Anthropic

Файл: `src/app/api/admin/estimates/generate/route.ts`

Проблема:

- до вибору моделі стоїть глобальна перевірка `GEMINI_API_KEY`
- якщо Gemini ключ відсутній, route падає навіть коли користувач обрав OpenAI або Anthropic

Наслідки:

- OpenAI і Anthropic сценарії фактично прив'язані до наявності Gemini
- продакшн-конфігурація стає крихкою

Що виправити:

- перевіряти лише ключ тієї моделі, яка реально обрана
- винести перевірку в окрему функцію по `model`

Пріоритет: Critical

### 4. У Gemini режимі не передається текст документів

Файл: `src/app/api/admin/estimates/generate/route.ts`

Проблема:

- `textParts` збираються з PDF/Excel/CSV
- але в `parts` для Gemini додається лише `prompt` і картинки
- отже Gemini не бачить текстову частину документів

Наслідки:

- основний AI сценарій за замовчуванням працює некоректно
- кошториси можуть генеруватися без урахування вхідних файлів

Що виправити:

- передавати в Gemini також `textParts.join("\n\n")`
- окремо протестувати:
  - тільки PDF
  - тільки Excel
  - PDF + image
  - image only

Пріоритет: Critical

## Важливі проблеми

### 5. Створення кошторису неатомарне

Файл: `src/app/api/admin/estimates/route.ts`

Проблема:

- спочатку створюється `estimate`
- потім окремо секційні item'и через цикл і `createMany`
- у разі падіння на середині процесу в БД залишається частково створений кошторис

Наслідки:

- биті або неповні кошториси в проді
- важке ручне прибирання даних

Що виправити:

- обгорнути весь create flow у справжню `prisma.$transaction`
- або створювати nested create повністю в одному запиті

Пріоритет: High

### 6. Номер кошторису через `count() + 1` дає race condition

Файл: `src/app/api/admin/estimates/route.ts`

Проблема:

- номер `EST-XXXX` рахується через загальну кількість записів
- паралельні запити можуть згенерувати однаковий номер

Наслідки:

- можливі конфлікти унікальності
- нестабільність під навантаженням

Що виправити:

- перейти на:
  - окрему sequence
  - окрему таблицю лічильника
  - UUID/cuid плюс окремий display number

Пріоритет: High

### 7. `next/image` дозволяє будь-який HTTPS домен

Файл: `next.config.ts`

Проблема:

- `hostname: "**"` відкриває image optimization майже для будь-якого хоста

Наслідки:

- зайвий security surface
- складніше контролювати джерела контенту
- може дати непередбачувану поведінку в проді

Що виправити:

- залишити whitelist лише потрібних CDN/доменів
- забрати wildcard host entirely

Пріоритет: High

### 8. Немає робочого quality gate перед деплоєм

Файли:

- `package.json`
- `tsconfig.json`

Проблема:

- `lint` зараз зламаний
- `tsc --noEmit` теж не зелений

Наслідки:

- деплой проходить без нормальної автоматичної перевірки
- баги потрапляють у продакшн без базового захисту

Що виправити:

- оновити lint script під актуальний Next 16 / ESLint flow
- налаштувати окремий test tsconfig або виключити тести з production typecheck
- додати CI pipeline:
  - lint
  - typecheck
  - build

Пріоритет: High

## Середні проблеми

### 9. Використовується deprecated `middleware` convention

Файл: `src/middleware.ts`

Проблема:

- build попереджає, що `middleware` deprecated і треба переходити на `proxy`

Наслідки:

- не ламає прод прямо зараз
- але накопичує технічний борг

Що виправити:

- мігрувати на актуальну convention для Next 16

Пріоритет: Medium

### 10. Landing page не використовує реальний CMS

Файл: `src/app/page.tsx`

Проблема:

- основний контент жорстко захардкоджений
- попри наявність CMS-сутностей у БД, публічний сайт не виглядає як сайт, що їх реально використовує

Наслідки:

- CMS у поточному стані частково "декоративна"
- редакторські зміни не впливають на публічний сайт автоматично

Що виправити:

- визначити, що саме має керуватись через CMS:
  - новини
  - портфоліо
  - сторінки
  - контактні блоки
- перевести публічні секції на читання з БД або CMS-layer

Пріоритет: Medium

### 11. AI routes працюють синхронно в межах одного HTTP запиту

Файли:

- `src/app/api/admin/estimates/generate/route.ts`
- `src/app/api/admin/estimates/refine/route.ts`

Проблема:

- довгі AI операції виконуються в request-response циклі
- немає черги, job status, retry або async orchestration

Наслідки:

- ризик таймаутів на Vercel
- нестабільність на важких документах
- поганий UX при повторних спробах

Що виправити:

- якщо AI-флоу стане production-critical:
  - винести в async jobs
  - додати статус задачі
  - зберігати результат генерації в БД
  - додати retry/backoff

Пріоритет: Medium

### 12. У seed є дефолтні паролі та масове видалення всіх даних

Файл: `prisma/seed.ts`

Проблема:

- використовується `password123`
- seed спочатку масово чистить усі таблиці

Наслідки:

- небезпечно для будь-якого середовища, окрім локального
- ризик помилкового запуску не в тому environment

Що виправити:

- явно заблокувати виконання seed у production
- додати перевірку `NODE_ENV !== "production"`
- змінити документацію, щоб seed ніколи не пропонувався для production

Пріоритет: Medium

## Низький пріоритет, але варто поліпшити

### 13. Забагато `any` у важливих ділянках

Де помітно:

- фінансові сторінки
- init routes
- estimates route

Що виправити:

- додати типи DTO для API request/response
- винести Zod-схеми
- зменшити неявні runtime-помилки

Пріоритет: Low

### 14. Частина логіки в API дублює доменні модулі

Наприклад:

- є `financial-workflow.ts`
- але частина фінансової логіки живе напряму в route handlers

Що виправити:

- зібрати доменну логіку в service layer
- залишити в route тільки auth, validation, response mapping

Пріоритет: Low

## Рекомендований порядок виправлень

### Етап 1. Security і доступ

1. Закрити ACL у всіх `/api/admin/*`
2. Закрити ACL у всіх `/api/projects/*`
3. Явно визначити права для `USER`
4. Додати helper для перевірки доступу на рівні роутів

### Етап 2. AI generation

1. Виправити перевірку ключів по вибраній моделі
2. Передати `textParts` у Gemini
3. Додати валідацію формату відповіді AI
4. Додати базові integration тести на generation route

### Етап 3. Data integrity

1. Переробити create estimate в транзакцію
2. Прибрати `count()+1` з генерації номера
3. Додати обробку rollback-сценаріїв

### Етап 4. Quality gates

1. Полагодити lint
2. Полагодити typecheck
3. Додати CI
4. Зробити build/lint/typecheck обов'язковими перед деплоєм

### Етап 5. Production hardening

1. Звузити image host allowlist
2. Заборонити production seed
3. Перейти з `db push` на міграції
4. Поступово винести важкий AI у background jobs

Це дасть найбільший ефект:

- зменшить ризик битих даних
- поверне контроль якості перед релізами

## Підсумок

Проєкт уже достатньо близький до робочого продакшну:

- збірка проходить
- архітектура зрозуміла
- основні бізнес-сутності вже реалізовані
- auth, Prisma і Vercel інтегровані коректно на базовому рівні

Але перед повноцінною production експлуатацією обов'язково треба закрити:

- доступ до API
- AI generation defects
- транзакційність створення кошторисів
- зламані перевірки якості

Без цього система може працювати, але з реальними ризиками витоку даних, неправильних AI-результатів і нестабільної підтримки змін.
